## 1. Yahoo stale-data fix — ✅ SHIPPED + LIVE-VERIFIED (do NOT redo)

**Real root cause (proven, not the handoff doc's theory):** Yahoo's OAuth **token-refresh** endpoint
(`api.login.yahoo.com`) **rejects Railway's datacenter IP** with a misleading `INVALID_CONSUMER_KEY`.
The *same* credentials refresh fine from a residential IP. The server can *use* an access token for
its 1h life but can never *refresh* one → data died ~1h after every paste. Not a hang, not throttling,
not credentials, not a code bug.

**Fix (shipped to `master`, deployed, ACTIVE):** a token **relay**. The owner's home mini PC refreshes
the token (residential IP), encrypts it (Fernet), and PATCHes it to a secret GitHub gist every 30 min;
the Railway scheduler pulls + decrypts it each cycle and writes it to the volume *before* reconnecting,
so the server never calls the blocked endpoint. All server-side relay code is dormant unless
`HEATER_TOKEN_RELAY_URL` + `HEATER_RELAY_KEY` are set (v1 byte-for-byte).

**Verified live (2026-06-07):** logged into the app, drove every page. Yahoo data is **LIVE** — League
Standings shows **"Yahoo: Live (via server)"** with Rosters/Standings/Matchup/FA/Transactions/Settings
all LIVE; Matchup Planner restored (Week 11 vs HUMAN INTELLIGENCE); My Team fresh. The spec's success
bar is met.
