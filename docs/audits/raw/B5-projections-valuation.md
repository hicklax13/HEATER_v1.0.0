# B5 — Projections / valuation (subagent result)

**Verdict:** SGP signs + rate-stat handling are correct and well-guarded everywhere (player_sgp/totals_sgp/total_sgp_batch/marginal_sgp all sign inverse cats L/ERA/WHIP correctly; rate stats volume-weighted never simple-averaged; LeagueConfig.inverse_stats includes L and is sole category source; Marcel faithfully implemented — 5/4/3 weights, reliability=PA/(PA+1200), age curves with inverse-stat flip, regression applied once; create_blended_projections self-protects vs blended-only sample data; VORP scarcity NOT double-counted into pick_score). **One HIGH at the foundation + one inverted blend.**

## Correctness findings
- **PV-C1** (L4, **HIGH**, conf 88) — *Ridge-stacking weights learned on a temporally-mismatched target.* `src/database.py:1089-1124` (`_load_stacking_weights`) + `projection_stacking.py:97-144`. Loads `actuals = season_stats WHERE season=2025` as the target, then loads each system's rows from the `projections` table — which has NO season/year column (schema database.py:159-198 + migrations 660-702) so it holds ONLY current 2026 forecasts. `compute_stacking_weights` regresses 2026 forecasts against 2025 outcomes; the learned weights are then applied to those same 2026 forecasts in `create_blended_projections` (database.py:1216-1225). Features (2026) and target (2025) describe different seasons → weights measure persistence, not forecast skill, and can over-weight whichever system most resembles last year's leaderboard. Conditional: only active when ≥10 rows 2025 season_stats AND ≥2 systems; else correctly falls back to uniform 1/n (database.py:1091-1106). Foundation every engine consumes. NOT a Known Design Choice (CLAUDE.md says weights "LEARNED"; projection_stacking.py docstring even says "regresses against prior-year actuals" — the wiring contradicts its own intent).
- **PV-C2** (L4, **Medium**, conf 72) — *Bayesian ROS Layer-2 reliability blend is INVERTED.* `src/bayesian.py:954-991` (`_blend_with_projection`): `result[stat] = reliability*marcel_val + (1-reliability)*proj_val`, reliability grows with accumulated PA/IP (threshold 1200 PA/400 IP). A fully-sampled veteran (reliability→1.0) is projected almost entirely from crude 3-year Marcel, with the expert ('blended' Steamer/ZiPS/DC) projection weighted toward 0; low-sample players lean on the expert. Backwards — experts already ingest full history + aging + park + role, strictly more informed. Reduces accuracy for exactly the players with the most data. Feeds update_ros_projections → ros_bayesian → load_player_pool (every ROS surface). Untested (no test exercises the reliability direction). NOT a Known Design Choice.
- **PV-C3** (L4, **Low**, conf 70) — Counting-stat stacking applied to volume columns (pa/ab/h/ip/er) with each column's OWN ridge weights, then rate stats recomputed from these independently-weighted components (`database.py:1212-1238`) → e.g. `h` weighted toward Steamer while `ab` weighted toward ZiPS yields an h/ab AVG no system produced. Volume columns should share a single playing-time weight set. Gated same as PV-C1 (often off); next-order concern after PV-C1.
- **PV-C4** (L4, **Low**, conf 62) — `compute_sgp_denominators` (`valuation.py:1004-1121`) simulates 12 teams by round-robin striping a globally-sorted pool (`vals[i::n]`) and takes np.std — near-perfectly even distribution understates true cross-team spread → biased-low denominator vs the documented "stddev of team totals." Three different denominator notions exist (static config / round-robin sim / standings median-gap); the advertised "Bayesian-updated denominators" is unimplemented (no denominator updater in bayesian.py). Only caller is ui_shared debug surface, so live impact limited.

## Enhancements (ranked)
1. **PV-E1** (High/Medium) — Persist per-system projections with a `forecast_season` key; learn stacking weights from a true held-out (prior-year forecast → prior-year actual) regression (fixes PV-C1; unlocks the marketed learned blend). Add `forecast_season INTEGER`; stop DELETEing prior seasons; build X from `projections WHERE forecast_season=Y` + y from `season_stats WHERE season=Y` for completed years, average weights, apply to current. Keep uniform fallback. Expected: lower projection RMSE/MAE vs uniform (skill-weighted ensembles beat naive avg by a few % on HR/K, 5-10 pts on rate stats).
2. **PV-E2** (Medium/Low) — Re-orient Bayesian Layer-2 so expert ∝ reliability, Marcel ∝ (1−reliability) — or treat Marcel as fallback/prior only (fixes PV-C2). Add a test asserting increasing total_weighted_vol moves the result TOWARD proj_val.
3. **PV-E3** (Medium/Medium) — Replace the 3 ad-hoc denominator estimators with one standings-stddev computation (Schechter SGP: denom = stddev of 12 season-end team totals), Bayesian-shrunk toward projected-roster-totals early-season; wire as the live denominator source (fixes PV-C4 + the doc/impl mismatch). Propagates into every SGP consumer (trade/FA/draft).

> Note: CLAUDE.md's "pick_score = SGP + scarcity + VORP" phrasing oversimplifies — scarcity_mult is a separate pool column consumed by waiver/fa/trade, NOT double-counted into value_all_players' pick_score (which uses VORP replacement scarcity). Consistent, just documented loosely.
