# B4 — Matchup & standings (subagent result)

**Verdict:** Core math sound — rate-stat aggregation correctly volume-weighted EVERYWHERE (AVG=ΣH/ΣAB, OBP component-summed, ERA=ΣER·9/ΣIP, WHIP=Σ(BB+H)/ΣIP); inverse-cat direction correct in all win-prob/sim paths; punt detection uses the required AND (gainable==0 AND rank>=10); MatchupContextService is the single 3-mode weight source; playoff_sim pins _PLAYOFF_SPOTS=4 + season_weeks=26. Findings concentrate in season-length constant drift, an un-invalidated derived cache (live stale-read), and a dormant momentum sign error.

## Correctness findings
- **MS-C1** (L5, **Medium**, conf 95) — *League Standings page uses `_TOTAL_WEEKS=24` instead of canonical 26.* `pages/6_League_Standings.py:74`. Used in synthetic-schedule fallback (`range(1,_TOTAL_WEEKS+1)` lines 789,1020) + magic-number horizon (`remaining=_TOTAL_WEEKS-_current_week+1` line 797). Lineup page correctly sources `_LC_W().season_weeks`(=26). Under-counts remaining matchups ~2 wks; truncates synthetic schedule when Yahoo schedule unavailable. NOT a Known Design Choice (plain drift; the Matchup Planner's week-relative horizon is a different justified case).
- **MS-C2** (L4, **Medium**, conf 88) — *`apply_time_decay` default `total_weeks=24` lets counting-stat time_fraction exceed 1.0 early-season (inflation, no clamp).* `src/trade_intelligence.py:87,107`. `time_fraction = weeks_remaining/max(total_weeks,1)`, weeks_remaining ∈[1,26] from canonical accessor → early season 25/24≈1.04, so counting SGP scaled >1.0 (amplified not decayed). Same 24/26 family at weekly_h2h_strategy.py:198,230; yahoo_data_service.py:462; `compute_schedule_urgency` hardcodes `if w>24: break` (trade_intelligence.py:181) ignoring wks 25-26. Default to season_weeks(26) + clamp time_fraction to [0,1].
- **MS-C3** (L4, **Medium**, conf 82) — *`standings_utils._cached_team_totals` module-global never invalidated in production → stale standings on long-lived process.* `src/standings_utils.py:19,75-76,136`. `get_all_team_totals()` returns the process-global whenever non-None + not force_refresh, short-circuiting BEFORE any YDS call. `clear_cache()` exists (line 140) but invoked only in tests — never in app.py/YDS. "Refresh All Data" calls `st.cache_data.clear()` + `YDS.force_refresh_all()` but neither touches this dict. **Acute under MULTI_USER/Railway:** single long-lived replica, scheduler rewrites standings each cycle, but every read-only member session that hit `get_all_team_totals` early keeps the first totals forever (YDS 30-min TTL bypassed). Consumed by Standings, Lineup, Trade Finder, Trade Analyzer. NOT a Known Design Choice (contradicts the live-first architecture).
- **MS-C4** (L4, **Low**, conf 90) — Momentum adjustment in `simulate_season_enhanced` has WRONG SIGN for inverse cats (ERA/WHIP/L). `src/standings_engine.py:417-423`: `adjustment=1.0+0.05*(mom-1.0)` applied uniformly; hot team (mom>1) RAISES ERA mean (=worse). **Currently dormant** — Standings page never passes `momentum_data` (no `momentum=` kwarg). Flagged because it's public API; future wiring would silently mis-rank pitching-heavy teams. Apply only to non-inverse, or invert for INVERSE_CATS.
- **MS-C5** (L5, **Low**, conf 70) — Standings category-rank grid ranks raw `get_standings()` teams with no `valid_teams` ghost filter. `pages/6_League_Standings.py:680-698`. Bug-D's `build_standings_totals(valid_teams=...)` wired only into evaluate_trade, not this page. Live Yahoo path reliably returns 12 teams so low risk; residual risk is SQLite-fallback serving a stale standings cache with a renamed team. `test_no_ghost_team_rank.py` covers only the engine path.
- **MS-C6** (**Info**, conf 99) — `playoff_sim.simulate_season()` docstring still says "top 6" after BUG-021 set spots to 4. `src/playoff_sim.py:307-308`. Code correct (`_PLAYOFF_SPOTS=4`). Doc-only.
