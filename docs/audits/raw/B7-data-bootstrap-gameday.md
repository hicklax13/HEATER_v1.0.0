# B7 — Data / bootstrap + game-day (subagent result)

**Verdict:** Silent-failure coverage is sound (every `_bootstrap_*` outer except writes refresh_log or re-raises; reconciliation + retry sound; check_staleness/freshness/connection_status/roster_fetch_is_complete/TWP-routing/IP-parse all HOLD). But TWO HIGH "engine built, wires never connected" data-quality bugs feed downstream engines stale/wrong data while CI stays green on synthetic test input.

## Correctness findings
- **DB-C1** (L5, **HIGH**, conf 95) — *Park-factor Tier 1 is permanently dead; emergency dict always served.* `src/data_bootstrap.py:608-652`. `_tier1_pybaseball()` returns `team_batting(year)` — a DataFrame of team BATTING stats, NOT a `{team: park_factor}` dict (not park-factor data at all). `fetch_with_fallback` returns `(DataFrame, 'primary')`; the next line `if isinstance(data, dict) and len(data)>0:` is ALWAYS False for a DataFrame → always falls to `_PARK_FACTORS_EMERGENCY_2026; tier='emergency'`. Park factors are PERMANENTLY the static 2026-04-18 hardcoded dict; Tier 1 can never run. The optimizer reads module-level `PARK_FACTORS` (=emergency dict) directly (`pages/2_Line-up_Optimizer.py:104`), so static values flow into every lineup/matchup park adjustment year-round. `test_sf22_park_factor_tier.py` only ever PATCHES `fetch_with_fallback` to return a dict — no test feeds the real `_tier1_pybaseball()` output, so CI is green while production never honors Tier 1. NOT a Known Design Choice (Data Sources table claims "Tier 1 honored when available"; production contradicts it; distinct from the FanGraphs-403 item — team_batting is a working endpoint, the bug is wrong fetcher + isinstance mismatch). **Corroborates nothing in handoff but is the highest-impact data-quality bug found.**
- **DB-C2** (L6→data-quality, **HIGH**, conf 90) — *Closer Monitor primary data path is dead.* `pages/3_Closer_Monitor.py:101-132` reads `st.session_state['closer_depth_data']` but NOTHING in src/ ever writes it (grep confirms read-only). Falls through to a heuristic picking the single highest-season-SV pitcher per team as "the closer" (confidence=sv/35). `_bootstrap_depth_charts` persists `depth_chart_role` to the players TABLE, not this session dict; `_bootstrap_game_day` returns only weather/pitchers/lineups/team_strength. So the `{team:{closer,setup,closer_confidence}}` dict `build_closer_grid` expects is never produced in prod. Heuristic systematically mis-ranks mid-season: setup man with 2-3 vulture saves outranks established closer; recently-installed closer with 1 SV shows ~0 confidence; committees collapse to one name. `test_closer_monitor*.py` feed a HAND-BUILT dict directly into `build_closer_grid` (validate in isolation; never receives that shape in prod). Page tells users data comes "from the FanGraphs depth chart data loaded at app launch" — false. NOT a Known Design Choice (closer accuracy explicitly in scope). **Corroborates handoff task #11.**
- **DB-C3** (L5, **Medium**, conf 80) — *Live-game matchup freshness gated at 30 min, not the advertised 5-min TTL.* `src/data_bootstrap.py:3266` `check_staleness('yahoo_data', yahoo_hours=0.5)` gates `_bootstrap_yahoo → _refresh_yahoo_aux → sync_all_team_matchups` (line 760); `yahoo_data_service.py:54` matchup TTL=300s. Scheduler ticks every 300s but per-team matchups only written inside the 30-min-gated `_bootstrap_yahoo`; `yahoo_matchup` has no independent short gate. Read-only members read the SQLite cache (no live client) → live H2H scores up to 30 min stale during games while UI implies 5-min freshness. Give `yahoo_matchup` its own ~5-min game-window gate. **Corroborates handoff task #12.**
- **DB-C4** (L4, **Low**, conf 78) — `fetch_game_day_weather` stamps `game_date` with UTC date (`game_day.py:345`) while schedule is fetched for the ET target date (`:288` get_target_game_date). Night games (7-10pm ET = 23:00-03:00 UTC) get a next-day weather row; forecast-hour index uses UTC hour into a local-time Open-Meteo array (second skew). Reuse ET target_date + `&timezone=` venue-local.
- **DB-C5** (L4, **Low**, conf 60) — `match_player_id` DNA-collision WARNING covers only the middle name-only fallback (`live_stats.py:126-147`); the trailing `LIKE '%lastname%' AND team=?` (153-162) and team-less `LIKE '% lastname'` (164-172) branches return without the warning. Team-less branch has NO team filter (only len==1 guard) → single-match-but-wrong-team possible for a caller that passed team_abbr. Route all fallbacks through the check or drop the team-less branch.
- **DB-C6** (L4, **Low**, conf 55) — team_strength rate reads use `_safe_float(x) or DEFAULT` (`game_day.py:691-692, 882-888`); fires on genuine 0.0 too. Harmless at team-aggregate scale (real 0.0 impossible for wRC+/OPS/ERA over a season) but a latent footgun if copied to player/single-game contexts. Use explicit None check.
