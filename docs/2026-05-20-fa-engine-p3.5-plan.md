## Executive summary

P1 fixed the root cause of the Crochet/Kirk bad-recommendation class (IL zeroing in `_roster_category_totals`). P2 made the engine industry-aligned (ROS+YTD blending, calibrated sustainability, multiplicative composite). But the live app still produces bad recommendations because the fixes were asymmetric or status-only:

| Surviving bug | Bad recommendation observed | Root cause |
|---------------|-----------------------------|------------|
| **(a) Asymmetric scarcity** | "Drop Jack Leiter (active SP) for Alejandro Kirk (3rd catcher, .150 AVG)" | `_positional_scarcity_factor` applied to ADDS via `_compute_base_value` but DROPS via `compute_drop_cost` are scarcity-blind. Backup catchers get over-boosted on the add side without paying scarcity on the drop side. |
| **(b) No roster-construction guard** | "Drop Eduardo Rodriguez (4W / 2.53 ERA, top SP) for Bryson Stott (.23 AVG)" — user only has 8 healthy SP | Engine doesn't check that the swap leaves a viable roster — happy to recommend the 3rd catcher when user has 2 catchers (one IL), or drop the 2nd SP when user has only 8 healthy starters. |
| **(c) Short-term unavailability treated as season-ending** | "Drop Framber Valdez (top SP, 6-day suspension ENDED 2026-05-20) for Jordan Westburg" | `_il_weight_from_status` returns 0.0 for "Suspended" / "NA" / "Restricted" / "OUT" unconditionally. Correct for season-ending statuses but catastrophically wrong for short suspensions. |

The engine correctly handled IL aces (Crochet protected) and produced one defensible swap (Seth Lugo's 1.40 WHIP for Cade Horton). The 3 remaining failures map cleanly to 3 surgical PRs.

**Acceptance criteria for P3.5**: user reviews 4 fresh FA recommendations on the live app and ≥3 of 4 are defensible per industry standard. If P3.5 doesn't get there, `systematic-debugging` the residual failures before continuing to original P3.

## PR15 — Symmetric positional scarcity on `compute_drop_cost`

**Files modified**:
- `src/waiver_wire.py` — extend `compute_drop_cost` signature to accept `replacement_levels: dict | None = None`. When provided, multiply `base_cost` by the per-position scarcity factor.
- `src/optimizer/fa_recommender.py` — `_score_drop_candidates` already runs after `_score_fa_candidates`. Hoist the computed `_replacement_levels` so both scorers see the same dict. Pass it through to `compute_drop_cost`.
- `src/valuation.py` — move `_positional_scarcity_factor` from `fa_recommender.py` to `valuation.py` as a public function `compute_positional_scarcity_factor(positions, replacement_levels)`. Both modules import from a shared location to avoid `fa_recommender` <-> `waiver_wire` circular import (`waiver_wire.py` is imported BY `fa_recommender.py`, so `waiver_wire` can't import scarcity from `fa_recommender`).
  - Keep the old private `_positional_scarcity_factor` in `fa_recommender.py` as a thin delegating wrapper for one cycle so existing tests keep importing it; mark it as a forwarding alias.

**Design choice (locked)**: dynamic scarcity from `compute_replacement_levels` (industry consensus — same path P1 PR3 took on the add side).

**TDD test (write first)**:
```python

def test_dropping_top_ss_costs_more_than_dropping_25th_of_equal_sgp():
    """The drop-side scarcity multiplier should make dropping a scarce-
    position player cost MORE than dropping a deep-position player with
    equivalent raw SGP. This pins symmetry with the add-side scarcity
    boost (P1 PR3) — otherwise the engine recommends draining scarce
    positions (catcher, SS) in favor of deep positions (OF, 1B)."""
    # Build a pool where:
    #   ss_player has positions="SS", raw SGP X
    #   of_player has positions="OF",  raw SGP X (same)
    #   replacement_levels = {"SS": low (scarce), "OF": high (deep)}
    # Expected: compute_drop_cost(ss_player, ..., replacement_levels=replacement_levels) >
    #           compute_drop_cost(of_player, ..., replacement_levels=replacement_levels)
```

Additional regression guards:
- `test_drop_cost_no_replacement_levels_unchanged.py` — When `replacement_levels=None`, return identical value to pre-PR15 (backward compatibility check).
- `test_drop_cost_neutral_position_unchanged.py` — A 1.0× scarcity factor (positions absent from `replacement_levels`) must return identical cost to the no-factor path.

**Risk**: P3 PR9 plans to migrate magic constants to `constants_registry`. The `_POSITIONAL_SCARCITY_MAX_BOOST = 0.25` cap should move with the function. Doing it now (in PR15) is fine — small, cohesive change.

## PR16 — Roster-construction guard

**Files modified**:
- `src/optimizer/fa_recommender.py` — new private function `_passes_roster_construction_guard(fa_data, drop_data, ctx) -> tuple[bool, str]`. Called inside `_evaluate_swaps` for each (add, drop) pair, after the IL/active matching gate. Returns `(False, reason)` to skip the swap and log the reason; `(True, "")` to allow.

**Hard rules (per user spec — locked)**:

1. **Add-side position cap**: count rostered players (excluding IL) eligible at each of the FA's positions. For each position, the cap is `Yahoo starting slots + 1` (one backup beyond starter). FA is rejected only if ALL of FA's eligible positions are at or above their cap.
   - Position caps per CLAUDE.md league config:
     - C=1+1=2, 1B=1+1=2, 2B=1+1=2, 3B=1+1=2, SS=1+1=2, OF=3+1=4
     - SP=2+1=3, RP=2+1=3
     - Util / P treated as flex slots (no independent cap) since they overlap many positions
   - Why "ALL positions at cap" not "ANY": a 2B/SS who's eligible at both positions could fill a hole at one even if the other is full. Per user spec example (3rd catcher), the issue was C-only FAs; this rule correctly catches it.

2. **Drop-side hitter floor**: never drop if it leaves user with fewer than **10 active hitters** (= starting lineup minimum: 1C+1×1B+1×2B+1×3B+1SS+3OF+2Util).
3. **Drop-side pitcher floor**: never drop if it leaves user with fewer than **8 active pitchers** (= starting lineup minimum: 2SP+2RP+4P).

These floors are anchored to the FourzynBurn starting-lineup count. Below these the user literally cannot field a full starting roster and would forfeit empty slots. IL players are EXCLUDED from the active count (they cannot start). BN players ARE included in the active count (they're flex healthy reserves).

**TDD tests (write first)**:
```python
# tests/test_fa_roster_construction_guard.py
def test_blocks_third_catcher_when_two_already_rostered():
    """User has Raleigh (IL) + Dingler (active) — both eligible at C only.
    Engine must not recommend a 3rd C-only FA even if that FA has higher
    composite score than Dingler. The 2-of-2 C-eligible roster is at cap."""

def test_allows_2b_ss_multi_position_when_one_position_full():
    """User has 2 SS-only rostered. FA is 2B/SS-eligible. 2B has 0 rostered
    (under cap). Engine must allow — the FA could fill the 2B hole."""

def test_blocks_drop_when_below_10_active_hitters():
    """User has exactly 10 active hitters + various pitchers. Drop candidate
    is a hitter. Engine must skip — the drop leaves 9 active hitters, below
    the 10-hitter starting-lineup minimum floor."""

def test_blocks_drop_when_below_8_active_pitchers():
    """User has exactly 8 active pitchers. Drop candidate is a pitcher.
    Engine must skip — the drop leaves 7 active pitchers, below the
    8-pitcher starting-lineup minimum floor."""

def test_construction_guard_allows_valid_swap():
    """Sanity check: a within-bounds swap (active 2B for FA 2B, user has
    plenty of buffer) passes the guard."""
```

**Design choice (locked)**: count IL players as "rostered" for position-cap purposes (user has Raleigh IL + Dingler at C → 2 catchers → at cap). Count IL players as "INACTIVE" for the drop-side hitter/pitcher floor (only active players count toward the floor).

## PR17 — Return-date-aware IL weighting

**Files modified**:
- `src/in_season.py`:
  - `_il_weight_from_status(status: str, expected_return_days: float | None = None) -> float` — extend signature. When status is in the "potentially short" set (`"SUSPENDED"`, `"NA"`, `"RESTRICTED"`, `"OUT"`, `"DTD"`) AND `expected_return_days` is provided, use the return-date curve instead of the string default.
  - `_roster_category_totals` — read `expected_return_days` from each row, pass to `_il_weight_from_status`.
- `src/optimizer/shared_data_layer.py`:
  - In `build_optimizer_context` (the section that calls `_get_il_return_dates`), merge return-date data INTO the roster DataFrame as a new column `expected_return_days` (float, days from now, or NaN). This is what `_roster_category_totals` reads.
  - Also surface Yahoo's news-derived suspension end-dates if available. The news_flags dict (`ctx.news_flags`) currently stores free-form text; PR17 extends that pipeline only if the text parses to a date (otherwise no-op, keep the news string). Out of scope: refactoring the news pipeline structure — that's bigger work.

**Return-date curve (locked)**:

```python
def _return_date_weight(expected_return_days: float | None) -> float:
    """Map days-until-return to a projection weight.

    Calibrated to match existing IL10/IL15/IL60 string defaults at
    canonical anchor points so the new return-date path is consistent
    with the pre-PR17 status-only path.

      < 0 days (already returned)  ->  1.0
       1 day                       ->  0.95
       7 days                      ->  0.85
      14 days                      ->  0.70  (anchors to IL10 default)
      21 days                      ->  0.55
      45 days                      ->  0.30
      60+ days OR None             ->  0.0   (fall through to status default)
    """
```

Interpolation: piecewise-linear between anchor points. Above 60 days → return 0.0 (treat as season-ending). When `expected_return_days is None` → fall through to the existing status string lookup (so existing IL10/IL15/IL60 strings keep working).

**TDD tests (write first)**:
```python

    """Framber Valdez scenario: status='Suspended', expected_return_days=1.
    Old behavior: weight=0.0 (any Suspended is season-ending).
    New behavior: weight ≈ 0.95 (1-day return is essentially active)."""

def test_long_suspension_returns_zero():
    """Indefinite suspension: status='Suspended', expected_return_days=None.
    Behavior: weight=0.0 (fall through to string default, unchanged)."""

def test_return_date_curve_anchors():
    """1d→0.95, 7d→0.85, 14d→0.70 (matches IL10 default), 60d+→0.0."""

def test_negative_days_returns_full_weight():
    """expected_return_days < 0 (return date in the past — player already
    back per ESPN) → weight=1.0 (treat as active)."""

def test_roster_totals_uses_return_date_column():
    """When the roster DataFrame has an expected_return_days column,
    _roster_category_totals must pass it to _il_weight_from_status."""

def test_roster_totals_backward_compat_no_return_date_column():
    """When the column is absent, behavior is unchanged from pre-PR17."""
```

**Risk**: ESPN return dates are best-effort. `_get_il_return_dates` returns a dict that might be sparse (only injured rostered players have entries, and only those ESPN tracks). For players without return dates, the function correctly falls through to the existing string-based weight — no regression. Yahoo suspension end-dates are surfaced via the news pipeline currently as free-form strings; PR17 will parse them if a date is detectable but otherwise no-op. If Yahoo's news field name is unclear, use `firecrawl` on the Yahoo Fantasy API docs (mentioned in resume prompt) — but observation: the existing alerts.py code already handles this via the news pipeline, so likely no new external research needed.

## Locked design decisions (do not re-litigate)

These were confirmed in the prior session and are not open for discussion:
- Dynamic scarcity from `compute_replacement_levels` (not fixed multipliers)
- Deprecate (don't delete) `waiver_wire.compute_add_drop_recommendations`
- Canonical blend weights: 0.70 ROS / 0.20 YTD / 0.10 L14
- Strict IL protection: never drop rostered IL players
- IL players count as "rostered" for position-cap purposes (PR16) but not toward the active-roster floor (PR16)
- Return-date weight curve anchored at IL10 default = 14-day return = 0.70 weight (PR17)
